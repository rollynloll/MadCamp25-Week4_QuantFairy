## ADR-001: 프로젝트 리팩토링

- **날짜**: 26.05.12
- **상태**: 확정

### 결정

1. 프로젝트를 전면 리팩토링한다.
   - 기존 몰입캠프에서 진행한 프로젝트는 oop 개념이 없는 상태에서 바이브 코딩에 의존하여 작성되어 구조가 불분명하고 확장성이 떨어진다. 따라서 ui는 남겨둔 채 내부 구조만 리팩토링하여 프로젝트 건전성을 유지한다.  
&nbsp;
2. 프로젝트 구조는 features / layers 두개의 층위로 분리하여 생각한다.
   - features는 실제 프로젝트의 기능별 분류이며 백테스트 / 자동매매 / 전략관리 로 큰 범주를 나눈다.
   - layers는 caller / engine / protocol 로 큰 범주를 나눈다.
&nbsp;
3. layers 각 계층의 의미는 다음과 같다.
   - caller: 유저의 진입점. 사용자의 요청을 받아서 engine을 호출하고 결과를 반환한다. web / cli 두 형태가 존재한다.
    caller
    ├── backend/   — HTTP 진입점 (라우터, 서비스, 에러 변환)
    └── cli/       — 터미널 진입점
   - engine: 순수 퀀트 로직만을 담는 엔진. db, api등의 정보를 완전히 배제해야 한다. feature에 따라 세개의 서브레이어로 나뉜다.
    engine
    ├── strategies/   — 전략이 무엇을 하는지 결정한다.
    ├── backtest/     — 전략 신호를 받아서 과거 데이터에 시뮬레이션한다.
    └── trading/      - 전략 신호를 실제 주문으로 변환한다.
   - protocol: engine과 infra 사이의 계약. engine은 구현체를 모르고, protocol interface만 알고 있다. 
    protocol
    ├── DataProvider   — price와 같은 시장 정보를 받는다.
    └── OrderExecuter  — 계좌 및 주문 정보를 받는다.

## ADR-002: 백테스트 유니버스 미변경

## ADR-003: cli db 저장 및 전략 crud 미루기

- **날짜**: 26.05.13
- **상태**: 확정

### 결정
cli caller 에서 산출되는 정보들을 json으로 저장하는데, 이를 웹 db에 동기화 하는 작업을 후순위로 미룬다. 
전략 crud 구현을 후순위로 미룬다.

### 이유
cli 에서 trading을 제대로 구현하는 것에 집중하기 위해서 웹과 관련된 요소가 이를 방해하는 것을 방지한다.
마찬가지로 전략 crud 구현을 위해서는 리소스가 많이 필요하므로 후순위로 미룬다.
